# 灵感扩展推文生成 Skill

## 技能概述

这个 skill 帮助你将零散的灵感（一句话、几个词、一个模糊的想法）扩展为**90 分+**的自媒体推文。

**目标定位：优质自媒体内容，不是技术文档**

它通过以下方式工作：
1. **框架内化** - 运用思考框架的思维设计问题，但不机械套用
2. **信息搜索整合** - 自动搜索高质量资料，补充相关数据和案例
3. **深度渐进提问** - 选择题定方向 + 开放问题丰满内容，**7-10 轮对话**直到完整
4. **迭代打磨优化** - 生成后反思优化，追求 90 分+ 质量
5. **多平台输出** - 生成适配不同平台的推文版本

**预期对话轮次：7-10 轮**
- 3-4 轮：获取方向和核心观点（技术文档水平）
- 5-7 轮：增加案例、情绪、独特性（自媒体水平）
- 7-10 轮：深度挖掘、反复打磨（优质自媒体水平）

## 核心机制：渐进式提问 + 框架内化

### 三个核心原则

**1. 框架内化，不外显**

运用思考框架的思维设计问题，但不说"我在用某某框架"：

```
❌ 错误："我推荐用 PREP 框架，现在问 Point：你的观点是什么？"
✅ 正确："你最想表达的核心观点是什么？"

问题带有框架思维，但对话自然。
```

**2. 选择题 + 开放问题结合**

```
选择题（AskUserQuestion）→ 快速锁定方向
开放问题（直接对话）      → 深度挖掘内容

选择题：骨架 | 开放问题：血肉
```

**3. 渐进深入，动态调整**

每轮问题基于上一轮的回答，自然过渡到下一个维度：

```
用户："todo 是被低估的命令"
    ↓
追问："被低估"具体指什么？（概念深挖）
    ↓
用户："大家不知道它是会话记忆机制"
    ↓
追问：解决了什么痛点？（价值追问）
    ↓
追问：有具体例子吗？（案例补充）
```

### 框架思维融入问题

| 框架思维 | 提问方式 | 示例 |
|----------|----------|------|
| Point | 核心观点是什么？ | "你想表达的核心观点是？" |
| Reason | 为什么这么认为？ | "为什么这么认为？理由是？" |
| What | 具体指什么？ | "这具体指什么？" |
| Why | 为什么重要？ | "解决了什么痛点？" |
| How | 怎么做到的？ | "怎么实现的？" |
| Example | 有例子吗？ | "有具体案例吗？" |
| Situation | 最初状态？ | "最初的状态是怎样的？" |
| Complication | 什么冲突？ | "出现了什么挑战？" |

**框架是思维工具，不是提问模板。**

### 信息搜索（多阶段执行，必须）

搜索必须在**多个阶段**进行，以确保收集到精准的外部佐证：

#### 第一阶段：初步背景搜索（提问开始前）

在提问开始前，执行初步信息搜索：
1. 识别主题中的核心概念
2. 构建多角度搜索查询
3. 使用 WebSearch 获取相关资料
4. 使用 WebFetch 深入阅读关键页面
5. 整理搜索结果供后续使用

#### 第二阶段：精准深度搜索（收集完核心观点后）

**这是最关键的一步** - 在收集完用户的核心观点、独特见解后，必须进行**第二轮精准搜索**：

1. **基于用户独特观点构建搜索词**
   - 提取用户提出的独特洞察/反常识观点
   - 用这些观点作为关键词进行反向搜索
   - 寻找支持或反驳的证据

2. **多角度验证搜索**
   - 搜索行业报告、新闻媒体
   - 搜索竞品/国外案例
   - 搜索数据支撑

3. **搜索失败时的处理**
   - 如果一次搜索失败/受限，尝试不同的关键词组合
   - 使用更通用或更具体的搜索词
   - 尝试英文搜索词获取国际视角
   - 记录搜索状态，告知用户（如因限制未能搜索）

#### 搜索关键词策略

| 阶段 | 搜索重点 | 示例关键词 |
|------|----------|------------|
| 初步搜索 | 背景信息、基本事实 | "千问3 发布"、"AI电商助手" |
| 精准搜索 | 用户独特观点验证 | "AI与电商冲突"、"流量分发 AI冲击" |
| 对比搜索 | 国际案例、竞品分析 | "ChatGPT 电商"、"国外AI购物助手" |

**搜索结果必须被整合到最终内容中，作为外部佐证。**

**即使搜索受限，也要尝试多次不同关键词，并将搜索状态告知用户。**

### 提问设计原则

**1. 基于主题追问**

问题必须紧扣用户输入的核心主题，不问无关问题。

**2. 框架内化设计**

运用框架思维设计问题，但不说"我在用某某框架"。

**3. 选择题 + 开放问题混合**

- **选择题**：有明确选项、需要快速分类时使用
- **开放问题**：需要故事/经历/情感/细节时使用
- **混合**：AskUserQuestion 的 "Other" 选项后继续追问

**4. 渐进深入，动态调整**

每轮基于上一轮回答，自然过渡到下一个维度。不是机械按模板走。

**5. 每轮聚焦 1-2 个问题**

避免信息过载，让用户有思考空间。

**6. 信息完整才停止**

检查核心维度，缺失什么追问什么，完整了才生成。

### 完整度判断标准（90 分+ 目标）

Skill 在判断信息是否完整时，检查以下维度：

#### 必须完整（缺失则继续追问，不可跳过）

| 维度 | 检查项 | 说明 |
|------|--------|------|
| **核心观点** | 要表达的核心观点是否清晰 | 文章的灵魂 |
| **目标受众** | 是否明确写给谁看 | 决定表达方式 |
| **发布平台** | 是否明确在哪里发 | 决定内容形式 |
| **真实案例** | 是否有具体的例子/经历 | **自媒体必须** |
| **情绪共鸣** | 是否有共鸣点/情绪钩子 | **自媒体必须** |
| **独特观点** | 是否有别人没提到的见解 | **90 分+ 必须** |

#### 应该完整（尽量追问）

| 维度 | 检查项 | 说明 |
|------|--------|------|
| **外部佐证** | 是否有搜索资料支撑 | 增加说服力 |

#### 最好完整（有则更佳）

| 维度 | 检查项 | 说明 |
|------|--------|------|
| **互动设计** | 是否有明确的行动召唤 | 引导读者参与 |
| **风格偏好** | 什么风格？ | 专业/幽默/故事化 |

---

### 90 分+ 内容的标准

| 分数 | 特征 | 缺失什么 |
|------|------|---------|
| **60-70 分** | 结构清晰、观点明确 | 缺少真实案例、情绪共鸣 |
| **80-85 分** | + 真实案例、情绪共鸣 | 缺少独特性、外部佐证 |
| **90 分+** | + 独特观点、外部佐证 | 无缺失，经过打磨 |

**只有当所有"必须完整"的维度都有时，才进入生成阶段。**

**真实案例、情绪共鸣、独特观点是自媒体内容的三大支柱，缺一不可。**

## 输出格式

### Markdown 格式规范（必须）

**所有推文内容必须使用标准 Markdown 格式输出**，包括：

| 格式元素 | 使用场景 | 示例 |
|----------|----------|------|
| **标题层级** | 主标题 H1、章节 H2-H4 | `# 标题` `## 章节` |
| **加粗强调** | 核心观点、关键词 | `**核心观点**` |
| **引用块** | 金句、重要论断 | `> 金句内容` |
| **列表** | 并列观点、步骤说明 | `- 项目1` |
| **分割线** | 区分不同部分 | `---` |
| **代码块** | 技术内容、数据 | ` ```代码``` ` |

**输出前检查清单**：
- [ ] 标题层级清晰（H1 主标题，H2 章节，H3 子章节）
- [ ] 核心观点用加粗强调
- [ ] 金句/重要论断用引用块
- [ ] 长内容用列表分点
- [ ] 章节之间用分割线区分
- [ ] 整体格式符合标准 Markdown 语法

## 最佳实践

1. **提供足够的上下文** - 即使是零散想法，尽量包含你关心的关键点
2. **诚实回答问题** - 交互式提问时的回答会直接影响推文质量
3. **明确目标平台** - 知道你要发在哪个平台，可以生成更合适的内容
4. **利用搜索结果** - 系统搜索的资料可以大大丰富你的内容
5. **多版本对比** - 对比不同版本的输出，选择最适合的
6. **迭代优化** - 可以基于生成的结果继续提问优化
